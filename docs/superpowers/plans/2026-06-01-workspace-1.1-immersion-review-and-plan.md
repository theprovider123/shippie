**Tested:** live prod (`shippie.app`, worker `cb4b1174`) + code at HEAD `5860fd5c`, via CDP at desktop 1440×900, tablet 768×1024, mobile 390×844; tools palate / tab / mise / chiwit / golazo.

## 1. The core diagnosis (evidence-backed)

The phased redesign got the *concept* right but the implementation still mixes **three worlds on one screen**: the marketing site chrome, the Workspace shell, and the embedded app's own chrome. The root cause is structural: **active tools are embedded panels, not owned canvases.** Shippie should behave like an OS layer that appears only when summoned; the app should be the place, not a thing inside a place.

**Measured (live), active tool's iframe as % of viewport:**

| Surface | Desktop 1440×900 | Mobile 390×844 |
|---|---|---|
| `/workspace?app=palate` (embedded) | **998×500 → 56% H, 69% W**, app top at 22% (pushed down by topbar+strip), page scrolls to 1157px | **352×500 → app starts at 479px = 57% down the viewport**, dock + rail present, page scrolls to 1692px |
| `/run/palate` (focused) | **1440×900 → 100% / 100%, top 0** | **390×844 → 100% / 100%, top 0** |

`/run` is already the north star. The embedded workspace mode is what betrays the idea.

**Code roots (HEAD):**
- `AppFrameHost.svelte` iframe: `width:100%; min-height:500px` — a hard 500px floor.
- `workspace/+page.svelte` `.viewport-area`: `min-height:500px; overflow:hidden` (clips, no scroll) and the `.workspace` grid is `align-content:start` (does **not** stretch the app to fill height).
- Non-focused shell = rail (`minmax(280px,360px)`) + topbar + resume strip + `.viewport-area`; on mobile the rail is `position:sticky` at top, so it **pushes the app down** and grows as more tools open.
- Focused branch (`{#if data.focused}`) = `.focused-shell{position:fixed;inset:0}` + iframe `100svh/100dvh` → full-bleed. **The fix is to make active-workspace render like focused.**
- `AppSwitcherGesture.svelte` is a **complete** edge-swipe + overlay-drawer + spring-anim + reduced-motion + first-run-hint system — but it's `gestureEnabled={false}` in focused mode. The reveal layer already exists; it's switched off.
- SDK already ships `useViewport()` (`width/height/svh/lvh/dvh/mode`), `useSafeArea()`, and a `shippie:host-insets` postMessage. Showcase apps are already responsive (clamp, dvh, safe-area, breakpoints). **The responsiveness foundation is there.**

## 3. The immersive model (north star)

**When a tool is open, Shippie becomes a latent ambient layer — on desktop and mobile alike.**

- The app owns 100% of the viewport (route active-workspace through the focused full-bleed presentation; no rail/topbar/dock taking layout space, no 500px floor, no clip).
- Shippie controls show briefly (~1.5s) on open, then **fade to quiet edge affordances**.
- Summon the shell by: **swipe from edge (mobile bottom / desktop left), edge-hover (desktop), ⌘K (desktop), or tapping the quiet handle.** The switcher **overlays and dims** the app (never pushes/resizes it), then fades away on selection or dismiss.
- This is mostly *re-enabling and polishing* `AppSwitcherGesture` (already built) + reusing the focused presentation (already built) — not new architecture.

## 4. User hats & edge cases the plan must hold

- **First-run visitor:** empty workspace → hero + starters (Phase 3, done). Must not auto-open a tool (done).
- **Returning power user, many tools open:** rail/switcher must not grow unbounded or push the app; Open list scrolls inside the overlay; warm-iframe cap (Phase F).
- **Mobile-on-the-go:** open tool = full screen, one-thumb summon, no dock inside tools.
- **Maker:** one responsive app, declared layout mode, screenshot gates (Phase E).
- **Accessibility:** focus trapping in the overlay (done for sheet), visible focus rings, ≥44px targets, reduced-motion respected (gesture already does), keyboard summon (⌘K) + Escape.
- **Sharing / connecting:** "Nearby/room" reframed as an explicit, contextual "Share / connect a device" surface, not ambient workspace noise.
- **Game / canvas app (golazo):** `fixedAspect` layout mode; full-bleed; safe-edges input region (API exists).
- **Mobile-designed app on a big screen:** `mobilePreferred` → host centers it in a phone-width stage instead of stretching to 998px.
- **Slow device:** freeze background tools, lazy lists, measure INP.

## 5. Mobile / desktop strategy (your explicit question)

**Recommendation: responsive-first single app + optional declared layout modes + a host viewport contract.** Do *not* make makers build separate mobile/desktop apps by default — that's a maintenance tax, and the SDK + showcases are already responsive-first.

**Maker declares one of (manifest `layout` field, new):**
- `responsive` *(default)* — adapts to host size.
- `mobilePreferred` — host centers it in a phone-width stage on large screens (fixes "mobile app strange on desktop").
- `desktopPreferred` — needs room; on small screens host offers a focused/scaled layout + a warning gate.
- `fixedAspect` — games/canvas declare an aspect ratio; host letterboxes.
- `immersive` — wants zero host chrome unless summoned (the default *feel* for all tools now, but explicit for canvas apps).

**Host → app viewport contract (extend the existing signals):** the SDK already has `useViewport()`/`useSafeArea()`/`host-insets`. Add host-pushed `mode`, `hostChromeVisible`, `inputType`, and density as CSS vars + an SDK event, so apps can adapt to *being summoned-over* and to the stage they're in.

**Maker gates:** screenshot checks at **390×844, 768×1024, 1440×900**, failing on horizontal overflow or clipped primary actions. (Hook into the existing deploy pipeline / `prove-offline` harness.)

## 6. Phased plan

### Phase A — Immersive active-tool hotfix (the unlock)
- Render `/workspace?app=<slug>` through the **focused full-bleed presentation** (the `/run` path): iframe owns the viewport, no rail/topbar/dock in layout, remove `min-height:500px` and `.viewport-area overflow:hidden`.
- On mobile, when a tool is active: hide the sticky rail, topbar, dock, footer. App = 100% from top.
- Keep `/run/<slug>` as the standalone/deep-link version (already full-bleed).
- **CDP acceptance:** active tool ≥ ~97% viewport H & W, top ≈ 0, no rail/dock above it, no h-overflow — at 390 / 768 / 1440.

### Phase B — Reveal switcher (latent shell)
- Re-enable `AppSwitcherGesture` in active-tool mode (`gestureEnabled=true`), overlay-not-push, edge='bottom' ≤640 / 'left' desktop.
- Auto-hide Shippie controls ~1.5s after open → quiet edge handle; re-summon on edge-hover (desktop), swipe (mobile), pointer-move, or **⌘K** (new). Escape/scrim/selection closes.
- Selecting a tool from the overlay closes it and swaps the canvas (no reload — open tools stay warm).

### Phase C — Chrome cleanup (one thing per screen)
- Active-tool top: a single compact command bar (tool name · status · the summon affordance), not site-nav + topbar + "Open in Shippie" + "Open in new tab". Demote/rename "Open in Shippie"; remove the duplicate "Home" square.
- Promote **Add / Search / Data / Access** to compact **icon affordances near the top** (keep rail foot as backup).
- Make **star / share / pin** high-contrast, labeled, ≥44px hit targets.
- Fix the "Start room" affordance: rename → **"Share nearby" / "Connect a device"**, fix the misaligned label/text, give it context.
- Resume/insight strip: one line, only when genuinely actionable (Phase 2 rule), and out of the way in immersive mode.

### Phase D — Home / dashboard cleanup
- Move **Updates** ("Stay / Update" receipts) into a badge → an **update inbox / tool-details** surface, off the main workspace. Rename: Stay → **Keep current**, Update → **Install update**.
- Move **Nearby / room** into a secondary **"Share / Nearby"** panel.
- **Rail order: Open (only when live) → Pinned → Recent → All.** Add an empty **"Pin tools here"** affordance when nothing's pinned; surface Recent above secondary utilities.

### Phase E — Maker viewport contract
- Add manifest `layout` modes (§5) + optional `aspectRatio`.
- Host pushes `mode` / `hostChromeVisible` / `inputType` / density (CSS vars + SDK event) on top of existing `useViewport`/`useSafeArea`/`host-insets`.
- Stage rendering: `mobilePreferred` → phone-width centered stage on desktop; `fixedAspect` → letterbox; `desktopPreferred` → small-screen warning/scaled layout.
- Screenshot gates at 390 / 768 / 1440 in the deploy pipeline.

### Phase F — Performance
- Cap warm iframes on mobile; freeze/suspend background tools after inactivity.
- Lazy-load the all-tools grid and insight/update data.
- Measure mobile INP / scroll before & after.

## 7. On the prior reviews + the warning cleanup
Both pasted reviews are directionally right and converge with this tested diagnosis; the "ambient OS layer / latent shell" framing is the correct north star and is adopted here. The earlier `5860fd5c` cleanup (switcher-sheet a11y + dead CSS) is valid but narrow — it does **not** touch any of the above; it's already committed and folds into the next deploy. **Do Phase A + B first** (canvas ownership + latent shell) — everything else becomes easier to judge once the app is the main character again.

## 8. Sequencing note
Treat A+B as a **Workspace 1.1 stabilization sprint** before any new features. Each phase: pure-logic TDD where applicable, CDP measurement gates (frame %, overflow, chrome-hidden), built in the worktree, FF-landed, deployed by the user. A is the unlock; C/D/E/F can follow in any order once A/B land.
